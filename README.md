# MINDCARE — веб-прототип

Платформа психологической поддержки студентов: психологи публикуют тесты, студенты проходят их и сами решают, показывать ли результат специалисту.

Это **прототип на React + Vite**, а не мобильное приложение. Он открывается в браузере и нужен, чтобы проверить логику продукта и показать её психологам, студентам и вузу до того, как писать Flutter-версию.

---

## Запуск

Нужен [Node.js 18+](https://nodejs.org).

```bash
npm install
npm run dev
```

Откроется `http://localhost:5173`.

Сборка для публикации:

```bash
npm run build     # результат в папке dist/
npm run preview   # посмотреть собранную версию локально
```

Папку `dist` можно выложить на Netlify, Vercel или GitHub Pages — бэкенд для этого не нужен.

## Как пройти весь сценарий за 5 минут

1. **«Я психолог»** → анкета (3 шага) → загрузите диплом (PDF, JPG или PNG до 3 МБ) → отправьте заявку.
2. Введите код подтверждения почты — в прототипе он показан прямо на экране.
3. Увидите экран «проверим в течение 24 часов» с обратным отсчётом. Тесты создавать пока нельзя — так и задумано.
4. Выйдите, войдите администратором → **Заявки** → откройте документ прямо в панели → **Подтвердить**.
5. Вернитесь под психологом → **Создать тест**: метаданные → вопросы с баллами → диапазоны результата → опубликовать.
6. Зарегистрируйте студента, пройдите тест, отправьте результат.
7. Под психологом откройте результат: ответы по вопросам, сумма баллов, приватная заметка, статус.

## Аналитика (BI)

Отдельная вкладка **📊 Аналитика**: у администратора — по всей платформе, у психолога — только по его тестам.

Что показывает:

- **Воронка** — открыли карточку → начали → дошли до конца → отправили психологу → психолог просмотрел. Видно, на каком шаге теряются студенты.
- **Доходимость** (completion rate) — главная метрика продукта, цель от 65%.
- **Динамика по дням** за 7, 14 или 30 дней.
- **Распределение по тяжести результата** — сколько студентов попало в спокойный, средний и высокий диапазон. Для психолога это важнее количества прохождений: показывает, находит ли платформа тех, кому нужна помощь.
- **Разрез по темам** и **таблица по тестам** со средним баллом и медианным временем прохождения.
- **Активные пользователи** за день, неделю и месяц, доля тех, кто прошёл второй тест.
- **Медиана времени до просмотра** результата психологом.
- **Выгрузка в CSV** — сводка, разрез по тестам, динамика. Открывается в Excel, имён студентов и их ответов в файлах нет.

Визуально: кольца прогресса для трёх ключевых долей, линейный график с заливкой, кольцевая диаграмма распределения, воронка с процентом потерь на каждом шаге, счётчики с анимацией набора числа.

Метрики считаются из двух источников: таблицы попыток (`attempts`) и журнала событий (`events`). Попытка создаётся в момент старта, а не завершения, — иначе брошенные на середине тесты не попали бы в статистику.

**Демо-данные.** При первом запуске генерируется около 120 прохождений за три недели, чтобы дашборд можно было показать. Они помечены флагом `demo`, вверху экрана есть переключатель «только реальные данные». Не показывайте эти цифры вузу как настоящие.

## Что работает

- Три роли: студент, психолог, администратор
- Подтверждение почты кодом при регистрации
- Загрузка документов (PDF, JPG, PNG) и просмотр их администратором
- Верификация психолога: подтвердить, запросить документы, отклонить с причиной; повторная подача
- Конструктор тестов: вопросы, баллы за варианты, диапазоны результата, черновик и публикация с валидацией
- Прохождение теста, подсчёт баллов, экран результата со шкалой всех диапазонов
- Явное согласие на отправку результата психологу и отзыв согласия
- Кабинет психолога: список результатов, ответы студента, заметка, статусы
- AI-чат с кризис-детектором и экраном экстренной помощи
- Админ: блокировка пользователей, скрытие тестов, журнал действий
- BI-дашборд с воронкой, динамикой, разрезами и выгрузкой в CSV

## Чего нет (и почему)

- **Настоящего бэкенда.** Всё лежит в `localStorage` браузера: у каждого устройства своя база, между людьми данные не ходят. Для демонстрации этого достаточно, для пилота — нет.
- **Безопасности.** Пароли хранятся в открытом виде, ролевые проверки только на клиенте. Так делать в проде нельзя.
- **Писем.** Код подтверждения показывается на экране, потому что отправка почты требует сервера.
- **Пуш-уведомлений.** Уведомления складываются в базу, но никуда не отправляются.

## AI-чат

По умолчанию чат отвечает подсказкой, что ключ не задан. Чтобы включить его локально:

```bash
cp .env.example .env
# впишите VITE_ANTHROPIC_API_KEY=sk-ant-...
npm run dev
```

Ключ попадает в код страницы и виден любому, кто откроет DevTools, поэтому так можно делать **только на своём компьютере**. В рабочей версии запрос к модели идёт через ваш бэкенд, а ключ остаётся на сервере.

## О визуальном стиле

Ориентир — Headspace, Calm, Notion Mobile, Linear. Правила, которых держится вся система:

- глубина создаётся тенью и слоями, а не рамками — бордеры почти прозрачные;
- радиусы крупные (18–26 px), внутренние отступы карточек 20 px;
- пастельная палитра с одним акцентом: лаванда ведёт, охра и мята поддерживают;
- движение мягкое — 200–400 мс, ease-out, без отскоков;
- шрифты: Manrope для заголовков и цифр, Inter для текста.

Палитра, тени и радиусы заданы переменными в начале `src/index.css`. Поменяв несколько значений, вы перекрасите всё приложение.

**Что анимировано:** появление экранов и карточек с задержкой по очереди, переезжающая «таблетка» активной вкладки, отрисовка колец и линий на графиках, счётчики цифр, прогресс прохождения теста, индикатор набора в чате, экран успеха после публикации теста, тост с рисующейся галочкой, скелетоны загрузки, pull-to-refresh, медленно дрейфующие пятна на фоне.

**Где анимаций намеренно нет.** Экран результата и кризисный экран оформлены тише остальных: пастельные пятна и «оживление» в момент, когда человек только что описал своё состояние как тяжёлое, читаются как обесценивание. Шкала результата анимируется только появлением сегментов, ничего не подпрыгивает. Если будете править дизайн, это стоит сохранить.

Вся система уважает `prefers-reduced-motion`: при включённой настройке анимации сокращаются до мгновенных.

### Переход между экранами

Сделан на keyed `motion.div`, а не на `AnimatePresence mode="wait"`. Второй вариант монтирует новый экран только после того, как доиграет уход старого, — и если анимация по любой причине не завершится, приложение застревает. Цена ошибки слишком велика, поэтому здесь новый экран появляется сразу, а анимация только на входе.
